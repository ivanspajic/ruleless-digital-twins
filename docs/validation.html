Model & Bindings Validation

WP2 provides a two-level validation mechanism: offline structural checks before startup, and optional live Home Assistant reachability checks on demand.

Level 1 — Offline CLI (V1)

Run before the MAPE-K loop. Requires no token, no running HA.

# Explicit path
dotnet run --project .\SmartNode\SmartNode\SmartNode.csproj ^^
  -- --validate-model .\config\ha-bindings.testlab.json

# Via environment variable
set HA_BINDINGS_FILE=.\config\ha-bindings.testlab.json
dotnet run --project .\SmartNode\SmartNode\SmartNode.csproj -- --validate-model

Exit codes: 0 = PASS or WARN  |  1 = FAIL  |  2 = bad invocation.

Checks performed: JSON shape and required fields, duplicate (sensorUri, procedureUri) keys, duplicate actuatorUri, reused haEntityId across actuators, haEntityId format (domain.object_id), known HA domains, actuator domain ↔ ActuatorKind consistency.

Level 2 — HTTP endpoint (V1.5 offline / V2 live)

Offline (default)

GET /api/model/validation

Returns the same structural report as the CLI. No token required. Response includes "checksLiveHomeAssistant": false.

{
  "mode": "offline",
  "source": ".\\config\\ha-bindings.testlab.json",
  "profile": "testlab",
  "result": "PASS",
  "errorCount": 0,
  "warningCount": 0,
  "sensors": 6,
  "actuators": 3,
  "errors": [],
  "warnings": [],
  "checksLiveHomeAssistant": false
}

Live (V2 minimal)

GET /api/model/validation?live=true

Runs the offline validator first, then checks the live HA runtime using HA_URL and TOKEN_HA.

Live checks: entity existence for every bound sensor/actuator; expected services for supported actuator domains.

If TOKEN_HA is absent, returns a clear failure without exposing secrets.

{
  "mode": "live",
  "source": ".\\config\\ha-bindings.testlab.json",
  "profile": "testlab",
  "result": "PASS",
  "errorCount": 0,
  "warningCount": 0,
  "sensors": 6,
  "actuators": 3,
  "haEntitiesChecked": 4,
  "haEntitiesReachable": 4,
  "haServicesChecked": 2,
  "haServicesReachable": 2,
  "errors": [],
  "warnings": [],
  "checksLiveHomeAssistant": true
}

HTTP status codes

CodeCondition
200PASS or WARN
400HA_BINDINGS_FILE is unset
422FAIL (hard validation error)
500Unexpected exception

Model health UI panel

The SmartNode chat UI (SmartNode/SmartNode/index.html) includes a Model health panel with two actions:

  • Refresh (default) — calls GET /api/model/validation (offline, no token required).
  • Live check — calls GET /api/model/validation?live=true and displays HA entity/service reachability counters.

The panel always shows checksLiveHomeAssistant: true/false explicitly so the operator knows which level of check was run.

Scope and limitations

  • V2 live validation does not check TTL ↔ bindings URI cross-consistency.
  • No soft-sensor dependency cycle detection yet.
  • No datatype normalisation pass yet.
  • V2 live checks require HA_URL and TOKEN_HA to be set.
  • No change to the MAPE-K loop.
